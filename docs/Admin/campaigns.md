# API админ-панели: Email-рассылки

**Базовый URL:** `/api/admin/campaigns`

**Авторизация:** все эндпоинты требуют авторизации администратора (`AdminGuard`). Запросы авторизуются через httpOnly cookie `adminToken`, которую браузер отправляет автоматически (заголовок `Authorization` также принимается для обратной совместимости).

Массовая отправка email-писем пользователям. Без Redis/Bull — используется простой фоновый процесс с трекингом прогресса в БД и батчингом (20 писем за раз, задержка 1 сек между батчами, чтобы не словить rate limit SMTP).

---

## Жизненный цикл кампании

DRAFT → SENDING → COMPLETED. Из SENDING возможен переход в FAILED.

- **DRAFT** — создана, но не запущена. Можно редактировать/удалять.
- **SENDING** — идёт рассылка, прогресс обновляется в `sentCount`/`failedCount` каждый батч.
- **COMPLETED** — все письма обработаны (успешно или с ошибкой, но процесс завершён).
- **FAILED** — неожиданная ошибка, процесс прерван. Можно запустить заново (`POST /:id/send`).

---

## Типы аудитории

| `targetType` | Кому отправлять | Unsubscribe-футер |
|--------------|-----------------|-------------------|
| `ALL` (default) | Всем зарегистрированным пользователям | Нет |
| `WITH_ORDERS` | Пользователям, у кого есть хотя бы один заказ | Нет |
| `WITHOUT_ORDERS` | Пользователям, кто ещё не покупал | Нет |
| `NEWSLETTER` | Подписчикам рассылки (см. [newsletter.md](./newsletter.md)) — только активным, с явным согласием | Да, добавляется автоматически |

Для маркетинговых рассылок используйте `NEWSLETTER` — у этих пользователей есть подтверждённое согласие (`acceptedTerms`) и они смогут отписаться по ссылке в письме (требование ФЗ 152 / GDPR). Типы `ALL` / `WITH_ORDERS` / `WITHOUT_ORDERS` не имеют согласия на маркетинг — используйте их только для сервисных уведомлений.

## 1. Создать кампанию

**POST** `/api/admin/campaigns`

Создаёт черновик. Отправку не начинает.

| Поле | Тип | Обязательное | Описание |
|------|-----|--------------|----------|
| `subject` | string | Да | Тема письма |
| `html` | string | Да | HTML-содержимое письма |
| `targetType` | enum | Нет | `ALL` (default) / `WITH_ORDERS` / `WITHOUT_ORDERS` / `NEWSLETTER` |

Ответ: объект кампании со статусом `DRAFT` и счётчиками `totalCount`, `sentCount`, `failedCount`.

---

## 2. Запустить рассылку

**POST** `/api/admin/campaigns/:id/send`

Запускает отправку в фоне. Возвращается немедленно — реальная отправка идёт асинхронно.

Ответ: `success: true`, сообщение и `totalRecipients`.

Ошибки: `400` — кампания уже в статусе `SENDING`/`COMPLETED`; `400` — нет получателей (пустая выборка).

Прогресс отслеживается опросом `GET /api/admin/campaigns/:id` — поля `sentCount` и `failedCount` обновляются каждые ~20 писем.

---

## 3. Список кампаний

**GET** `/api/admin/campaigns`

| Параметр | Тип | По умолчанию | Описание |
|----------|-----|--------------|----------|
| `page` | number | 1 | Номер страницы |
| `limit` | number | 20 | Размер страницы |

Ответ: объект с полями `campaigns` (массив с `subject`, `status`, `totalCount`, `sentCount`, `failedCount`, `startedAt`, `completedAt`, `createdAt`) и `pagination`.

---

## 4. Получить кампанию

**GET** `/api/admin/campaigns/:id`

Полная информация, включая `html` и текущий прогресс. Используется для построения прогресс-бара.

---

## 5. Удалить кампанию

**DELETE** `/api/admin/campaigns/:id`

Ограничение: нельзя удалить активную кампанию (`status: SENDING`) — сначала дождитесь завершения.

---

## Технические детали

- **Батчинг:** отправка группами по 20 писем с паузой 1 секунда между батчами (снижает риск попадания в спам-листы SMTP). Константы `BATCH_SIZE` и `BATCH_DELAY_MS` — в `src/admin/campaigns/campaigns.service.ts`.
- **Падение сервера во время SENDING:** кампания останется в статусе `SENDING` с сохранённым прогрессом; можно запустить повторно через `POST /:id/send` — рассылка начнётся заново (адреса не дедуплицируются между запусками, возможны дубли).
- Для продакшна с большими объёмами (>10 000 писем) рекомендуется вынести в BullMQ с Redis.
