# API промокодов (публичное)

API для работы с промокодами в магазине. Позволяет валидировать промокоды и применять скидки к заказам.

**Базовый URL:** `/api/promo`

**Авторизация:** Опциональна (работает для гостей и авторизованных пользователей). Для авторизованных токен берётся из httpOnly cookie `accessToken` (отправляется браузером автоматически); заголовок `Authorization: Bearer <token>` также принимается для совместимости.

## Валидация промокода

**POST** `/api/promo/validate`

Проверяет валидность промокода и рассчитывает скидку.

**Авторизация:** Опциональна (работает для гостей и авторизованных).

### Поля запроса:

| Поле | Тип | Обязательное | Описание |
|---|---|---|---|
| `code` | string | да | Код промокода |
| `orderAmount` | number | нет | Сумма заказа для проверки минимального порога |
| `cartItems` | object[] | нет | Товары в корзине для расчёта скидки |

Каждый элемент `cartItems`:

| Поле | Тип | Обязательное | Описание |
|---|---|---|---|
| `productId` | number | да | ID товара |
| `quantity` | number | да | Количество |
| `price` | number | да | Цена за единицу |
| `cardStatus` | string | нет | Статус карточки (`NEW` для новинки) |
| `title` | string | нет | Название товара (для текста ошибки про новинки) |

### Поля ответа (валидный промокод):

| Поле | Тип | Описание |
|---|---|---|
| `isValid` | boolean | `true` для валидного промокода |
| `discount` | number | Размер скидки в рублях (для FREE_DELIVERY — `0`) |
| `isFreeDelivery` | boolean | Присутствует и `true` для типа FREE_DELIVERY |
| `message` | string | Человекочитаемое описание скидки |
| `promoCode` | object | Данные промокода |

Поля объекта `promoCode`:

| Поле | Тип | Описание |
|---|---|---|
| `id` | number | ID промокода |
| `code` | string | Код |
| `type` | string | `PERCENTAGE` / `FIXED` / `FREE_DELIVERY` |
| `value` | number | Значение (процент или сумма; для FREE_DELIVERY — `0`) |
| `description` | string | Описание |
| `isFreeDelivery` | boolean | Признак бесплатной доставки (для FREE_DELIVERY) |

### Поля ответа (невалидный промокод):

| Поле | Тип | Описание |
|---|---|---|
| `isValid` | boolean | `false` |
| `discount` | number | `0` |
| `message` | string | Причина невалидности |

### Возможные причины невалидности:

| Сообщение | Причина |
|-----------|---------|
| `Промокод не найден` | Такого промокода не существует |
| `Промокод не активен` | Промокод деактивирован администратором |
| `Промокод начнёт действовать DD.MM.YYYY` | Промокод ещё не начал действовать |
| `Срок действия промокода истек` | Промокод истёк по дате `validUntil` |
| `Промокод больше не действителен` | Достигнут максимальный лимит использований (`maxUses`) |
| `Промокод доступен только для зарегистрированных пользователей` | Гость пытается использовать промокод с `requiresAuth=true` |
| `Промокод недоступен для вас` | Промокод доступен только определённым пользователям (персональный) |
| `Вы уже использовали этот промокод максимальное количество раз` | Достигнут лимит на пользователя (`maxUsesPerUser`) |
| `Минимальная сумма заказа: XXXX ₽` | Сумма заказа меньше `minOrderAmount` |
| `Промокоды не применяются к заказам с новинками (Название товара)` | В корзине есть товар с меткой NEW, промокод имеет `excludeNewItems=true` и не является FREE_DELIVERY |
| `Промокод не применим к товарам в корзине` | Нет подходящих товаров (все исключены или не входят в `specificProductIds`) |

## Применение промокода в заказе

Промокод применяется автоматически при создании заказа, если передать поле `promoCode` (строка с кодом) в теле запроса.

**POST** `/api/orders`

Поле запроса, относящееся к промокоду:

| Поле | Тип | Обязательное | Описание |
|---|---|---|---|
| `promoCode` | string | нет | Код промокода |

### Поведение:

- Промокод валидируется на сервере.
- Если невалиден — возвращается ошибка `400` с причиной.
- Если валиден — скидка применяется автоматически.
- Использование записывается в историю.

### Поля ответа заказа, связанные с промокодом:

| Поле | Тип | Описание |
|---|---|---|
| `subtotal` | number | Сумма товаров до скидки |
| `discountAmount` | number | Размер применённой скидки |
| `deliveryPrice` | number | Стоимость доставки (для FREE_DELIVERY — `0`) |
| `total` | number | Итог к оплате |
| `promoCode` | string | Применённый код |
| `promoCodeId` | number | ID промокода |

Помимо этого, у каждой позиции `items[]` присутствует поле `discountAmount` — размер скидки на конкретную позицию.

## Логика работы промокодов

### Типы скидок:

1. **PERCENTAGE** — процентная скидка (10% = `value: 10`).
2. **FIXED** — фиксированная скидка в рублях (500₽ = `value: 500`).
3. **FREE_DELIVERY** — бесплатная доставка (скидка на доставку).

### Ограничения применения:

#### По товарам:

- **appliesTo: ALL** — применяется ко всем товарам.
- **appliesTo: PRODUCT** — только к определённым товарам (`specificProductIds`).
- **excludedProductIds** — исключённые товары (не применяется).

#### По новинкам:

- По умолчанию PERCENTAGE и FIXED **не применяются** к товарам с меткой NEW (если админ не отключил эту проверку через `excludeNewItems`).
- FREE_DELIVERY всегда применяется к любым товарам, включая новинки.

#### Лимит на количество товаров:

- **maxItems** — скидка применяется только к N самым дорогим товарам. Пример: `maxItems=2` → скидка на 2 самых дорогих товара в корзине.

### Ограничения по использованию:

| Параметр | Описание |
|---|---|
| `maxUses` | Общее количество использований промокода |
| `maxUsesPerUser` | Сколько раз один пользователь может использовать |
| `minOrderAmount` | Минимальная сумма заказа в рублях |
| `validFrom` / `validUntil` | Период действия промокода |
| `requiresAuth` | Требуется авторизация (`true` = только для зарегистрированных) |
| `allowedUserIds` | Список UUID пользователей (пустой = для всех, заполненный = персональный) |

### Приоритет применения:

1. Проверяется активность и срок действия.
2. Проверяются лимиты использований.
3. Проверяется доступность для пользователя.
4. Проверяется минимальная сумма заказа.
5. Проверяются товары (наличие новинок, исключения).
6. Рассчитывается применимая сумма.
7. Применяется скидка.

## Примеры использования

### Пример 1: Простая процентная скидка на всё

Промокод даёт 20% скидку на все товары. Корзина: товар 1 (5000₽ × 1) и товар 2 (2500₽ × 2 = 5000₽).

- Товар 1: 5000 × 20% = 1000 ₽ скидка.
- Товар 2: 5000 × 20% = 1000 ₽ скидка.
- **Итого скидка: 2000 ₽.**

### Пример 2: Фиксированная скидка

Промокод даёт фиксированную скидку 500 ₽. Корзина на 8000 ₽.

- Фиксированная скидка: 500 ₽.
- **Итого к оплате: 7500 ₽.**

### Пример 3: Скидка на N самых дорогих товаров (maxItems)

Промокод: `type=PERCENTAGE`, `value=30`, `maxItems=2`. Корзина: товар 10 (8000₽), товар 11 (5000₽), товар 12 (3000₽).

1. Сортировка по цене: 8000, 5000, 3000.
2. Скидка к топ-2: товар 10 → 2400 ₽, товар 11 → 1500 ₽; товар 12 без скидки.
3. **Итого скидка: 3900 ₽.**

### Пример 4: Скидка на конкретные товары

Промокод: `type=FIXED`, `value=500`, `appliesTo=PRODUCT`, `specificProductIds=[20,21,22]`. Корзина: товар 20 (куртка, 9500₽), товар 30 (5500₽).

- Товар 20 подходит, товар 30 — нет.
- Скидка применяется только к товару 20: **500 ₽.**

### Пример 5: Скидка с исключениями (excludedProductIds)

Промокод: `type=PERCENTAGE`, `value=15`, `appliesTo=ALL`, `excludedProductIds=[10,11,12]`. Корзина: товар 5 (8000₽), товар 10 (5000₽).

- Товар 5: 8000 × 15% = 1200 ₽ скидка.
- Товар 10: исключён.
- **Итого скидка: 1200 ₽.**

### Пример 6: Бесплатная доставка

Промокод FREE_DELIVERY при заказе от 3000 ₽. Корзина на 4500 ₽.

- Скидка на товары не применяется (`discount = 0`), в ответе `isFreeDelivery: true`.
- Стоимость доставки становится 0 ₽ при оформлении заказа.
- Работает с любыми товарами, включая новинки.

### Пример 7: Ошибка — промокод не применим к новинкам

В корзине товар с меткой NEW, промокод имеет `excludeNewItems=true`. Результат: `isValid: false`, сообщение «Промокоды не применяются к заказам с новинками (Название товара)».

### Пример 8: Ошибка — минимальная сумма заказа

Промокод требует минимум 5000 ₽, корзина на 3000 ₽. Результат: `isValid: false`, сообщение «Минимальная сумма заказа: 5000 ₽».

### Пример 9: Ошибка — лимит использований исчерпан

Достигнут `maxUses`. Результат: `isValid: false`, сообщение «Промокод больше не действителен».

### Пример 10: Промокод только для зарегистрированных пользователей

Промокод с `requiresAuth=true`.

- Попытка гостя (без авторизации) → `isValid: false`, «Промокод доступен только для зарегистрированных пользователей».
- Попытка авторизованного (токен в httpOnly cookie отправляется автоматически) → скидка применяется. Для 15% при заказе 5000 ₽: скидка 750 ₽, к оплате 4250 ₽.

### Пример 11: Полный flow создания заказа с промокодом

1. **Шаг 1 — проверка промокода:** **POST** `/api/promo/validate` с `code`, `orderAmount`, `cartItems` → ответ `isValid: true`, `discount`, `message`.
2. **Шаг 2 — создание заказа:** **POST** `/api/orders` с данными покупателя, `items` и полем `promoCode` → в ответе `subtotal`, `discountAmount`, `deliveryPrice`, `total`, `promoCode`, `status`.

## Сложные сценарии и комбинации

### Сценарий 1: maxItems + excludedProductIds

Промокод: `type=PERCENTAGE`, `value=25`, `maxItems=3`, `excludedProductIds=[50]`. Корзина: товар 10 (10000₽), товар 50 (8000₽, исключён), товар 20 (6000₽), товар 30 (4000₽).

1. Товар 50 исключается.
2. Остаются товары 10, 20, 30.
3. Берутся топ-3 из оставшихся — все три.
4. Скидка: 2500 + 1500 + 1000 = **5000 ₽.**

### Сценарий 2: Несколько товаров одного типа (quantity > 1)

Промокод: 20% на топ-2 позиции. Корзина: товар 100 (5000₽ × 3), товар 200 (8000₽ × 1).

1. Каждый товар — отдельная позиция (несмотря на quantity).
2. Сортировка по цене за единицу: 8000, затем 5000.
3. Скидка: товар 200 → 1600 ₽; товар 100 → (5000 × 3) × 20% = 3000 ₽.
4. **Итого скидка: 4600 ₽.**

### Сценарий 3: Фиксированная скидка больше стоимости товара

Промокод даёт 1000 ₽, товар стоит 800 ₽. Скидка ограничивается стоимостью товара: **800 ₽** (не 1000 ₽).

### Сценарий 4: Различия между requiresAuth и allowedUserIds

- `requiresAuth: false`, `allowedUserIds: []` → любой может использовать (гости и зарегистрированные).
- `requiresAuth: true`, `allowedUserIds: []` → только авторизованные пользователи, но любые.
- `allowedUserIds: ["uuid1","uuid2"]` → только пользователи с конкретными UUID.

Приоритет проверки: сначала `requiresAuth`, затем `allowedUserIds`.

### Сценарий 5: Персональный промокод (allowedUserIds)

Промокод с `allowedUserIds=["<uuid>"]`.

- Попытка гостя → `isValid: false`, «Промокод недоступен для вас».
- Попытка авторизованного с совпадающим `userId` → скидка применяется.

### Сценарий 6: specificProductIds + новинки

Промокод: `type=PERCENTAGE`, `value=20`, `appliesTo=PRODUCT`, `specificProductIds=[10,20,30]`, `excludeNewItems=true`. Корзина: товар 10 (NEW, 5000₽), товар 20 (4000₽).

- При `excludeNewItems=true` → `isValid: false`, «Промокоды не применяются к заказам с новинками».
- При `excludeNewItems=false` → скидка: товар 10 → 1000 ₽, товар 20 → 800 ₽, итого **1800 ₽.**

## Детальный расчет скидок

### Алгоритм применения PERCENTAGE и FIXED

**Шаг 1: Фильтрация товаров**

1. Исключить товары из `excludedProductIds`.
2. Если `appliesTo === PRODUCT`, оставить только товары из `specificProductIds`.
3. Если `excludeNewItems === true`, исключить товары с `cardStatus='NEW'`.

**Шаг 2: Сортировка и выбор товаров**

1. Отсортировать оставшиеся товары по цене (по убыванию).
2. Если указан `maxItems`, взять первые N товаров.

**Шаг 3: Расчёт скидки**

- Для **PERCENTAGE**: скидка = сумма по позициям `(price × quantity × value / 100)`.
- Для **FIXED**: скидка = `min(value, сумма (price × quantity))`.

**Пример PERCENTAGE:** позиции 5000×2 и 3000×1, скидка 20%: (10000 × 0.20) + (3000 × 0.20) = 2000 + 600 = **2600 ₽.**

**Пример FIXED:** позиции на 8000 ₽, скидка 1500 ₽: `min(1500, 8000)` = **1500 ₽.**

### Алгоритм для FREE_DELIVERY

1. Валидация только по общим параметрам (активность, даты, лимиты).
2. Проверка `minOrderAmount`.
3. Установка флага `isFreeDelivery: true`.
4. Стоимость доставки обнуляется при создании заказа.

FREE_DELIVERY не зависит от товаров в корзине, новинок или исключений.

## Частые вопросы (FAQ)

### 1. Можно ли использовать несколько промокодов одновременно?

Нет. Система поддерживает только один промокод на заказ.

### 2. Что происходит, если промокод стал невалидным между проверкой и созданием заказа?

При создании заказа промокод валидируется повторно. Если он стал невалидным — вернётся ошибка `400` с объяснением причины (поля `statusCode`, `message`, `error`).

### 3. Учитывается ли скидка по промокоду при расчёте минимальной суммы заказа?

Нет. Проверка `minOrderAmount` происходит **до применения скидки**. Пример: промокод требует от 5000 ₽ и даёт 20%; корзина на 5000 ₽ — промокод применяется, к оплате 4000 ₽.

### 4. Как работает скидка на товары с разными размерами?

Каждый размер — отдельный item в корзине. Товар с одним `productId` в двух размерах (M и L) считается двумя позициями. Промокод с `maxItems=1` применится только к одной позиции (самой дорогой / первой по порядку).

### 5. Промокод на «2 самых дорогих товара» — это 2 единицы или 2 позиции?

2 позиции (независимо от quantity). С `maxItems=2` для корзины из товара 1 (1000₽ × 5) и товара 2 (3000₽ × 1) скидка применится к обеим позициям: позиция 2 → 600 ₽, позиция 1 → 1000 ₽.

### 6. Можно ли применить промокод к заказу, где все товары — новинки?

Если промокод имеет `excludeNewItems=true` (по умолчанию) и тип PERCENTAGE или FIXED — нет. Если тип FREE_DELIVERY — да, бесплатная доставка применится.

### 7. Что означает «Промокод не применим к товарам в корзине»?

Эта ошибка возникает, когда: промокод имеет `appliesTo=PRODUCT` со `specificProductIds`, и в корзине нет ни одного товара из списка; либо все подходящие товары исключены через `excludedProductIds`.

### 8. В чём разница между requiresAuth и allowedUserIds?

- `requiresAuth: true` — промокод доступен всем зарегистрированным пользователям; гости не могут; конкретные UUID не нужны.
- `allowedUserIds: [...]` — промокод доступен только конкретным пользователям; нужно знать их UUID заранее (персональные/VIP промокоды).

### 9. Как узнать, сколько раз пользователь уже использовал промокод?

Эта информация доступна только администраторам через **GET** `/api/admin/promo-codes/:id/stats`. Для клиента она скрыта; при превышении лимита вернётся `isValid: false` с сообщением «Вы уже использовали этот промокод максимальное количество раз».

## Интеграция на фронтенде

### Рекомендуемый flow:

1. Пользователь вводит промокод.
2. Вызвать **POST** `/api/promo/validate` (опционально, для предварительной проверки).
3. Показать результат: при `isValid` — «Промокод применён! Скидка N ₽»; иначе — сообщение с причиной из поля `message`.
4. Добавить `promoCode` в запрос создания заказа.
5. Обработать возможную ошибку (если промокод стал невалидным между проверкой и созданием).

### Упрощённый flow (без предварительной проверки):

1. Пользователь вводит промокод.
2. Сразу добавляем код в запрос создания заказа.
3. Если ошибка — показываем причину.

### Рекомендации по запросу validate с фронта:

- Отправлять `code` (в верхнем регистре), `orderAmount` (текущая сумма корзины) и `cartItems` с полями `productId`, `quantity`, `price`, `cardStatus`.
- Использовать debounce (например, 500 мс) при автопроверке по мере ввода, начиная с 3 символов.
- При смене корзины перевалидировать ранее применённый промокод.
- Отменять предыдущий запрос (AbortController) при быстрой смене кода, чтобы избежать гонок.

| Невалиден | Красный | Текст причины (из `message`) | Показать поле для нового кода |
| FREE_DELIVERY | Зелёный | «Бесплатная доставка активирована!» | Показать кнопку удаления |

### Отображение скидки в итогах заказа

- Обычная скидка: показать строку «Промокод КОД» с суммой скидки со знаком минус, затем пересчитанный «Итого».
- FREE_DELIVERY: показать стоимость доставки как «Бесплатно (КОД)».

### Информация о промокоде (tooltip / раскрывающийся блок)

Можно показывать тип скидки, область применения и срок действия.

### Обработка edge cases

- Промокод введён, но корзина пуста → не давать применить, показать «Добавьте товары в корзину перед применением промокода».
- Промокод валиден, но корзина изменилась → автоматически перевалидировать.
- Промокод стал невалидным к моменту оформления → при ошибке `400` создания заказа показать причину и предложить обновить промокод.
- Автоматическое преобразование кода в верхний регистр при вводе.

### Сохранение промокода

- Можно сохранять последний валидный код (например, в localStorage) и автозаполнять при загрузке.
- Поддержать чтение кода из URL-параметра (для маркетинговых кампаний, например `?promo=SALE20`).

### Аналитика

Рекомендуется отправлять события «промокод применён» и «ошибка применения» в систему аналитики (код, размер скидки, сумма заказа, текст ошибки), чтобы понимать популярность промокодов и частые ошибки.

## Коды ошибок

### HTTP статус-коды

| Код | Описание | Когда возникает |
|-----|----------|----------------|
| 200 | OK | Промокод успешно валидирован |
| 400 | Bad Request | Промокод невалиден (см. `message` в ответе) |
| 401 | Unauthorized | Промокод требует авторизации (`allowedUserIds` не пустой) |
| 500 | Internal Server Error | Ошибка на сервере при валидации |

### Сообщения об ошибках (message)

#### Промокод не найден или неактивен

| Сообщение | Причина | Решение |
|-----------|---------|---------|
| `Промокод не найден` | Промокода с таким кодом нет в БД | Проверить правильность ввода |
| `Промокод не активен` | `isActive = false` | Промокод деактивирован админом |

#### Ограничения по времени

| Сообщение | Причина | Решение |
|-----------|---------|---------|
| `Промокод начнёт действовать DD.MM.YYYY` | Текущая дата < `validFrom` | Дождаться начала действия |
| `Срок действия промокода истек` | Текущая дата > `validUntil` | Использовать другой промокод |

#### Лимиты использования

| Сообщение | Причина | Решение |
|-----------|---------|---------|
| `Промокод больше не действителен` | `usedCount >= maxUses` | Общий лимит исчерпан |
| `Вы уже использовали этот промокод максимальное количество раз` | Превышен `maxUsesPerUser` | Пользователь исчерпал свой лимит |

#### Ограничения доступа

| Сообщение | Причина | Решение |
|-----------|---------|---------|
| `Промокод доступен только для зарегистрированных пользователей` | `requiresAuth: true` и пользователь не авторизован | Войти в аккаунт или зарегистрироваться |
| `Промокод недоступен для вас` | `allowedUserIds` не пустой и не содержит `userId` текущего пользователя | Промокод персональный |

#### Ограничения по заказу

| Сообщение | Причина | Решение |
|-----------|---------|---------|
| `Минимальная сумма заказа: XXXX ₽` | `orderAmount < minOrderAmount` | Добавить товаров на нужную сумму |
| `Промокоды не применяются к заказам с новинками (Название товара)` | В корзине товар с `cardStatus='NEW'` и `excludeNewItems=true` | Убрать новинку или использовать другой промокод |
| `Промокод не применим к товарам в корзине` | Нет подходящих товаров (все исключены или не входят в `specificProductIds`) | Добавить подходящие товары |

## Безопасность и best practices

### 1. Валидация на стороне клиента (необязательная)

Перед отправкой запроса можно добавить базовые проверки: код не пустой; длина от 3 до 50 символов; допустимы только буквы, цифры и дефисы. Это не заменяет серверную валидацию, а только улучшает UX.

### 2. Не доверять клиентским данным

Скидку и итоговую сумму рассчитывает только сервер. Не передавать на сервер вычисленный на клиенте `total` — отправлять `items` и `promoCode`, а итог брать из ответа сервера.

### 3. Обработка race conditions

При быстрой смене промокода отменять предыдущий запрос (AbortController), чтобы не показать устаревший результат.

### 4. Логирование и аналитика

Логировать применение промокода и ошибки валидации; отправлять соответствующие события в аналитику. Это помогает понять, какие промокоды популярны, какие ошибки встречаются чаще и на каком этапе пользователи отказываются от заказа.
